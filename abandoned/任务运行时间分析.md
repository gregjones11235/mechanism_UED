# 任务运行时间分析（Oscar 实测）

> **目的**：记录 Stage 1 / Stage 2 在 Oscar 上的**真实**单 run 墙钟、吞吐（sps）、并发配额，作为后续 Stage 3 耗时估算与排队规划的唯一可信地基。
>
> **数据源**：`sacct`（精确 Elapsed）+ `~/logs/minimax/*/logs.csv`（tick / sps）+ 作业 `.out` 头部（配置）。全部经 ssh 直连 Oscar 拉取，**非文档转抄、非估计**。
>
> **生成日期**：2026-06-19。
>
> ⚠ **本文档专治"文档里的瞎猜数字"**：此前 `STAGE1_运行流程.md` / `STAGE1_作业并发实测.md` 把 ACCEL 的 ~1h 单 run 当成所有阶段通用，导致 Stage 2 耗时被严重低估。本文档以 sacct 实测为准，纠正所有此类表述。

## 0. 一句话结论

- **Stage 1 ACCEL（plr runner, n_students=1）≈ 1h/run**——这是文档里"~1 小时"的真实出处，**仅对 ACCEL 成立**。
- **Stage 1 PAIRED（paired runner，修复 hidden_dim bug 后）≈ 4.1h/run**——这才是 **Stage 2 的正确对照基准**（Stage 2 继承 PAIRED）。
- **Stage 2 multi_teacher：N=1 ≈ 3.7h、N=2 ≈ 7h、N=4 ≈ 14h**；吞吐 `sps` 大致 `∝ 1/N`（N 每翻倍，吞吐减半，墙钟翻倍）。
- **N=1 ≈ PAIRED（甚至略快）= 正常**：multi_teacher 在 N=1 退化成单 teacher，开销与 PAIRED 同量级。增长纯粹来自 N 个 teacher 的 rollout/评估，**不是代码异常**。
- **有效并发 = 2 GPU**（QOSMaxGRESPerUser 硬卡），与单 run 时长共同决定整套墙钟。

## 1. ⚠ 关键勘误：Stage 2 的基准是 PAIRED，不是 ACCEL

| 方法 | runner | n_students | 单 run 墙钟（实测） | 是否 Stage 2 基准 |
|---|---|---|---|---|
| **ACCEL** (Stage 1) | `plr` | 1 | **~1:02:xx** | ❌ **不是**（血统不同） |
| **PAIRED** (Stage 1) | `paired` | 1 | **~4:09:xx** | ✅ **是**（Stage 2 继承 PAIRED） |
| Stage 2 multi_teacher N=1 | `multi_teacher`（继承 PAIRED） | 2 | **~3:45:xx** | — |

> **为什么这条勘误重要**：
> - 若错拿 ACCEL（1h）当基准 → Stage 2 N=1 的 3.7h 看起来"慢 3.6×"，会误判 multi_teacher 有异常开销、白白去查代码。
> - 按正确的 PAIRED（4.1h）基准 → Stage 2 N=1（3.7h）**与 PAIRED 同量级甚至略快**，开销完全正常，增长全部来自 N。
> - 根因：ACCEL 走 `plr` runner 且 `n_students=1`；PAIRED 与 multi_teacher 同血统。`oscar_minimax_stage2.sh` 注释本就写明"对标基线：PAIRED（N=1 退化点）"。

### 3.1 吞吐随 N 的标度（外推 Stage 3 的关键斜率）

| N | sps（约） | 相对 N=1 | 单 run 墙钟（约） |
|---|---|---|---|
| 1 | 34,000 | 1.0× | 3.7h |
| 2 | 17,000 | 0.50× | 7h |
| 4 | 9,400 | 0.28× | 14h |

→ **`sps ∝ 1/N`（近似线性反比）**：N 每翻倍，吞吐减半、墙钟翻倍。
→ 来源：每个 update step 要为 N 个 teacher 各跑一批关卡生成 + argmax 还要对 N 批各评一次（`vmap` 并行但仍占算力）；`n_students=2` 是固定额外开销（相对 PAIRED 的 n_students=1）。
→ ⚠ 此斜率是**同质 PVL** 下的。Stage 3 的 learnability/CENIE estimator 比 PVL 重、non-IC 机制有额外开销，**此斜率是下界，Stage 3 实际更慢**——具体外推待 Stage 3 代码定型后单独评估。

## 4. 并发配额与整套墙钟

- **有效并发 = 2 GPU**（`QOSMaxGRESPerUser`）：array 提交 10 seed **不是同时跑**，而是两两成对、每隔"单 run 时长"放行下一对。
- **整套墙钟 ≈ ⌈seed数 / 2⌉ × 单run时长 + 排队**。例：PAIRED 10 seed ≈ 5 波 × 4.1h ≈ 20h（不含跨作业排队）。
- **跨作业接力排队**：第二套作业要等第一套腾出 2 张卡才开跑（Stage 1 实测 PLR 排队 5h09m 才轮到）。
- ⚠ `STAGE1_作业并发实测.md` 用"单 run≈1h"估 Stage 2 的 120 run（得 60–110h），**低估**：Stage 2 单 run 是 3.7–14h，按 2 GPU 串行，全量 120 run 实际远超此数——这也是 Stage 2 应降级为 sanity、不跑全量的算力理由之一。

### 4.1 绕开 2 GPU 瓶颈的两条路（未验证，待 Stage 3 评估时再定）

1. **多 seed vmap**：1 个作业内 vmap 跑多 seed，把"需要的卡数"从 N 降到个位数，绕开 2 卡配额。需改代码。
2. **申请更高 GRES 配额**：向 CCV 申请提高 `QOSMaxGRESPerUser`，无需改代码，未必批。

## 6. Stage 4 阶段 1（go/no-go）—— 真实墙钟（实测）

- **规模**：`TOTAL_TIMESTEPS=1e8`（短跑）= **15 eval epoch**；`NUM_ENVS=256`、`ROLLOUT_STEPS=1000`、
  `NUM_STEPS=512`、`EVAL_FREQ=50`、`HIDDEN_SIZE=512`、`GEN_NUM_LEVELS_PER_GEN=64`、N=3 generator。
  **注意：阶段 1 用的 NUM_ENVS/ROLLOUT_STEPS 已是设计文档 §2.2 说的"真实规模"**，阶段 1→2 唯一变的是
  `TOTAL_TIMESTEPS`（1e8→3e8）。
- **数据源**：每 run 的 `_wandb.runtime`（含编译），35/35 run 全 `_step=750` 跑满。每张卡一次 1 run（N=3 单 run 占满一卡）。

| 档位 | GROUP_NAME | n | 单 run 墙钟 min~max | 均值 | /epoch |
|---|---|---|---|---|---|
| 基线 | s4-base | 7 | 18.2~29.3 min | **27.4 min** | ~106 s |
| explore (λ=5.0) | s4-lambda-5_0 | 7 | 17.3~28.2 min | 26.4 min | ~102 s |
| 甜区 (λ=1.0) | s4-lambda-1_0 | 7 | 16.5~29.5 min | 22.0 min | ~85 s |
| exploit (λ=inf) | s4-lambda-inf | 7 | 16.6~29.5 min | 25.3 min | ~98 s |
| fallback (λ=none) | s4-lambda-none | 7 | 16.5~28.1 min | 19.9 min | ~76 s |

- **关键事实（反直觉，纠正设计文档 §4 的担忧）**：N=3 generator 注入档单 run 墙钟 ≈ 基线（~23 min vs 27 min），
  **并未比 N=1 明显更慢**。原因：`jaxnav_sfl.py:905` 的"拆 jit 提速"已把 N=3 generator 段抽成独立编译单元
  （否则单 seed 编译 >8h），编译开销摊薄后 N=3 的额外算力只占小头。设计文档 §4"CPU 上 25min 编不完一个 epoch"
  是 **CPU 无拆 jit** 的旧状况，**GPU + 拆 jit 后不成立**。
- **编译开销**：`[DIAG]` 实测 eval_step=0=164s（含编译）vs eval_step=1=113s（纯运行）→ 一次性编译 ≈ 50s，
  占 15-epoch 总墙钟的小头（~4%）。
- **墙钟波动**（16~30 min 跨 seed）来自共享节点 GPU 调度差异，非配置不稳定；规划取均值，排队规划取上界。

## 7. Stage 4 阶段 2（真实规模 SOTA 对标）—— 推算墙钟

> **推算方法**（非瞎猜）：阶段 1→2 **只**把 `TOTAL_TIMESTEPS` 1e8→3e8（15→45 eval epoch，设计文档 §1.5
> 实测线性换算），其余 config 不变。故单 run 墙钟 ≈ `编译(50s) + per_epoch × 45`，per_epoch 取阶段 1 实测。
> 这是**对实测的线性外推**，比 minimax 那种跨血统外推可信得多。

### 7.1 阶段 2 单 run 墙钟（推算）

| 档位 | per_epoch(实测 s) | 阶段 2 单 run（45 epoch） | 小时 |
|---|---|---|---|
| 基线 s4-base | 106 | ~80 min | **1.34 h** |
| λ=5.0 | 102 | ~78 min | 1.29 h |
| λ=1.0 | 85 | ~64 min | 1.07 h |
| λ=inf | 98 | ~74 min | 1.24 h |
| λ=none | 76 | ~58 min | 0.97 h |

- **注入档单 run 典型 ≈ 1.1 h，保守上界（用阶段 1 最慢 run 1771s 外推）≈ 1.45 h**。
- ⚠ **阶段 2 可能比线性外推略慢的一个风险**：difficulty estimator 激励 generator 造更难的图，阶段 2 长训下
  invalid/病态图率可能上升（见记忆 `stage4-stall2-place-start-goal-unbounded-whileloop`），更频繁触发
  `MAX_PLACE_TRIES` 上界 → 每 epoch 略慢。已加 ×1.3 余量吸收。

### 7.2 阶段 2 整套墙钟（**4 GPU 并行 = 2 人 × 2 卡**）

> 算法：GPU·h = Σ(run 数 × 单 run h)；墙钟 = GPU·h / 4；×1.3 吸收波次取整 + 跨作业排队。
> seed = 10（对齐 baseline rliable 口径，设计文档 §2.1）。

| 阶段 2 矩阵方案 | run 数 | GPU·h | 4 卡墙钟 | +排队余量(×1.3) |
|---|---|---|---|---|
| **最小**（base + λ=1.0 + λ=inf，3 档×10seed） | 30 | 36.5 | 9.1 h | **≈ 12 h（0.5 天）** |
| 含 none 消融（4 档×10seed） | 40 | 46.2 | 11.5 h | **≈ 15 h（0.6 天）** |
| 全档（5 档×10seed） | 50 | 59.1 | 14.8 h | ≈ 19 h（0.8 天） |

### 7.3 ⭐ 结论：阶段 2 用 2×2=4 GPU 并行预计多久

- **推荐方案（最小，base+λ=1.0+λ=inf，30 run）**：**约 12 小时（半天多）** 跑完阶段 2 训练。
- 若把 none 消融也纳入训练矩阵（40 run）：**约 15 小时**。
- **这只是训练墙钟**；阶段 2 还要 ① 对每个 final checkpoint 跑 CVaR + hand-designed test set + 100-map
  评测（评测比训练轻，单 checkpoint 分钟级，见阶段 1 eval epoch ~49s 纯运行），② 整理 rliable 表。
  评测 + 制表保守再加 **半天**。
- **端到端阶段 2 ≈ 1 天**（训练 12h + 评测/制表 ~半天，4 GPU 并行）。比 Stage1/2 minimax 时代（PAIRED 单套 20h、
  multi_teacher 全量上百 h）**快一个量级**——因为 jaxnav N=3 拆 jit 后单 run 仅 ~1 h，且只跑 30 run 不是上百。

## 8. Stage 4 数据可信度声明

- 单 run 墙钟来自 35 个 finished run 的 `_wandb.runtime`（含编译），已剔除 4 个 crashed 残留。
