# 任务运行时间分析（Oscar 实测）

> **目的**：记录 Stage 1 / Stage 2 在 Oscar 上的**真实**单 run 墙钟、吞吐（sps）、并发配额，作为后续 Stage 3 耗时估算与排队规划的唯一可信地基。
>
> **数据源**：`sacct`（精确 Elapsed）+ `~/logs/minimax/*/logs.csv`（tick / sps）+ 作业 `.out` 头部（配置）。全部经 ssh 直连 Oscar 拉取，**非文档转抄、非估计**。
>
> **生成日期**：2026-06-19。
>
> ⚠ **本文档专治"文档里的瞎猜数字"**：此前 `STAGE1_运行流程.md` / `STAGE1_作业并发实测.md` 把 ACCEL 的 ~1h 单 run 当成所有阶段通用，导致 Stage 2 耗时被严重低估。本文档以 sacct 实测为准，纠正所有此类表述。

## 0. 一句话结论

- **Stage 1 ACCEL（plr runner, n_students=1）≈ 1h/run**——这是文档里"~1 小时"的真实出处，**仅对 ACCEL 成立**。
- **Stage 1 PAIRED（paired runner，修复 hidden_dim bug 后）≈ 4.1h/run**——这才是 **Stage 2 的正确对照基准**（Stage 2 继承 PAIRED）。
- **Stage 2 multi_teacher：N=1 ≈ 3.7h、N=2 ≈ 7h、N=4 ≈ 14h**；吞吐 `sps` 大致 `∝ 1/N`（N 每翻倍，吞吐减半，墙钟翻倍）。
- **N=1 ≈ PAIRED（甚至略快）= 正常**：multi_teacher 在 N=1 退化成单 teacher，开销与 PAIRED 同量级。增长纯粹来自 N 个 teacher 的 rollout/评估，**不是代码异常**。
- **有效并发 = 2 GPU**（QOSMaxGRESPerUser 硬卡），与单 run 时长共同决定整套墙钟。

## 1. ⚠ 关键勘误：Stage 2 的基准是 PAIRED，不是 ACCEL

| 方法 | runner | n_students | 单 run 墙钟（实测） | 是否 Stage 2 基准 |
|---|---|---|---|---|
| **ACCEL** (Stage 1) | `plr` | 1 | **~1:02:xx** | ❌ **不是**（血统不同） |
| **PAIRED** (Stage 1) | `paired` | 1 | **~4:09:xx** | ✅ **是**（Stage 2 继承 PAIRED） |
| Stage 2 multi_teacher N=1 | `multi_teacher`（继承 PAIRED） | 2 | **~3:45:xx** | — |

> **为什么这条勘误重要**：
> - 若错拿 ACCEL（1h）当基准 → Stage 2 N=1 的 3.7h 看起来"慢 3.6×"，会误判 multi_teacher 有异常开销、白白去查代码。
> - 按正确的 PAIRED（4.1h）基准 → Stage 2 N=1（3.7h）**与 PAIRED 同量级甚至略快**，开销完全正常，增长全部来自 N。
> - 根因：ACCEL 走 `plr` runner 且 `n_students=1`；PAIRED 与 multi_teacher 同血统。`oscar_minimax_stage2.sh` 注释本就写明"对标基线：PAIRED（N=1 退化点）"。

### 3.1 吞吐随 N 的标度（外推 Stage 3 的关键斜率）

| N | sps（约） | 相对 N=1 | 单 run 墙钟（约） |
|---|---|---|---|
| 1 | 34,000 | 1.0× | 3.7h |
| 2 | 17,000 | 0.50× | 7h |
| 4 | 9,400 | 0.28× | 14h |

→ **`sps ∝ 1/N`（近似线性反比）**：N 每翻倍，吞吐减半、墙钟翻倍。
→ 来源：每个 update step 要为 N 个 teacher 各跑一批关卡生成 + argmax 还要对 N 批各评一次（`vmap` 并行但仍占算力）；`n_students=2` 是固定额外开销（相对 PAIRED 的 n_students=1）。
→ ⚠ 此斜率是**同质 PVL** 下的。Stage 3 的 learnability/CENIE estimator 比 PVL 重、non-IC 机制有额外开销，**此斜率是下界，Stage 3 实际更慢**——具体外推待 Stage 3 代码定型后单独评估。

## 4. 并发配额与整套墙钟

- **有效并发 = 2 GPU**（`QOSMaxGRESPerUser`）：array 提交 10 seed **不是同时跑**，而是两两成对、每隔"单 run 时长"放行下一对。
- **整套墙钟 ≈ ⌈seed数 / 2⌉ × 单run时长 + 排队**。例：PAIRED 10 seed ≈ 5 波 × 4.1h ≈ 20h（不含跨作业排队）。
- **跨作业接力排队**：第二套作业要等第一套腾出 2 张卡才开跑（Stage 1 实测 PLR 排队 5h09m 才轮到）。
- ⚠ `STAGE1_作业并发实测.md` 用"单 run≈1h"估 Stage 2 的 120 run（得 60–110h），**低估**：Stage 2 单 run 是 3.7–14h，按 2 GPU 串行，全量 120 run 实际远超此数——这也是 Stage 2 应降级为 sanity、不跑全量的算力理由之一。

### 4.1 绕开 2 GPU 瓶颈的两条路（未验证，待 Stage 3 评估时再定）

1. **多 seed vmap**：1 个作业内 vmap 跑多 seed，把"需要的卡数"从 N 降到个位数，绕开 2 卡配额。需改代码。
2. **申请更高 GRES 配额**：向 CCV 申请提高 `QOSMaxGRESPerUser`，无需改代码，未必批。
