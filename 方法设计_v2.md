## 0. 为什么有 v2：v1 的三个被推翻点（用户压力测试的结果）

v1 的实现（已在 job 3582377 跑过一次，因 GPU OOM 在 seed training 阶段崩溃，未跑到 auction）暴露三个与用户 idea 不一致的地方，逐一被用户点破：

1. **【推翻】三个 proposer 用同一个 DiCode 原版 prompt，只靠底座异质**。用户 idea = 不同 LLM proposer 设计关卡时**侧重点不一样**（主动的角色分化），不是同 prompt 硬撑。→ v2 引入 **persona 分化的 proposer**。
2. **【推翻】feasibility 该不该是 proposer**。用户初想"ambitious + feasibility 都当 proposer"。分析后：ambitious=生产者（造关），feasibility=评判者（判断适不适合）＝**两种工种**。硬凑成对等 proposer 会让 auction 选择拧巴。→ v2 把 feasibility 的**把关**职能归 **Critic**，但保留一个**求稳 proposer** 在生产层（见下）。
3. **【推翻】只加求稳 proposer 不够，auction 会系统性淘汰求稳关**。用户点破：Coverage（depth 加权，深层权重 8×）+ AmbitionGain（深度×缺口）两项都偏向"深/难"，求稳关（浅层、适配当下）三项全垫底 → 永远选不上 → 求稳 proposer 形同摆设。根因 = **depth 被 double-count，且 bid 里没有任何一项奖励"适合 student 现在学"**。→ v2 加**第四项 bid = Learnability**（奖励适配当下，与 depth 对冲）。

## 1. v2 总体架构（3 角色 + 四项 bid）

> **注**：曾设计 4 角色（含 Critic-Feasibility），但 2026-07-01 代码核实判定 Critic 冗余不实现（§2.4）。实际架构 = **3 persona proposer + 四项 bid**。

```
DiCode generation cycle：①选parent → ②生成描述【我改这里】→ ③生成代码 → ④编译
                                        │
  ┌─────────────────────────────────────┴──────────────────────────────────────┐
  │ ②描述层 v2                                                                    │
  │                                                                              │
  │ 生产层（Proposers，都在"造关"，但侧重点 persona 分化；型号按特长配）：           │
  │   • Proposer-Ambitious (DeepSeek) : 攻长链条前置 / 深层依赖 + scaffold（往难）  │
  │   • Proposer-Feasible  (GLM)      : 造 student 能力边缘、刚够得着的关（往稳）    │
  │   • Proposer-Breadth   (Qwen)     : 铺多样技能类别（往广，读 archive 类别覆盖）  │
  │        ↓ 各自独立造关（independent，不 debate，不共享中间结果）                  │
  │   ~N_parents × 3 个候选提案                                                    │
  │        ↓ barrier                                                              │
  │ bid 信号（四项，全部模/次模，不破坏 (1-1/e)）：                                  │
  │   1. Coverage    (次模，代码算) : 互补覆盖，深层成就权重高（也否决"不新颖"）      │
  │   2. Endorsement (模，LLM互评)  : 其他 proposer 中立三维评分（可解/对准/有用）    │
  │   3. AmbitionGain(模，代码算)   : 深度 × target 缺口                            │
  │   4. Learnability(模，代码算)   : 父关官方 p(1-p)，适配当下、与depth对冲         │
  │        ↓                                                                       │
  │ auction 选 top-k（贪心，四项加权）→ 送 ③代码层                                   │
  └──────────────────────────────────────────────────────────────────────────────┘
```

**难 / 稳 / 广三个方向在生产层就平衡**（不会全造难关），Learnability bid 在选择层给求稳关撑腰，Coverage 保互补、压重复。**生产层供给 + 选择层奖励**配套。把关（不可解/太难/不新颖）由 Learnability + Coverage 两项 bid 自然覆盖，无需独立 Critic（§2.4）。

## 2. 四个角色的职责（prompt 设计的依据，prompt 本身待单独设计+审核）

### 2.1 Proposer-Ambitious（生产，往难）
- **侧重**：识别 target 里 student 还没解锁的**长链条 / 深层依赖**（如"先造火把照明→才能深挖→才能采钻石"），主动造攻这些前置的关。
- **产出倾向**：depth tier 3-4 的成就、多步前置、late-game 目标。
- **护栏**：仍须可解（起终点/资源可达），激进 ≠ 不可解。

### 2.2 Proposer-Feasible（生产，往稳）★用户核心补充
- **侧重**：判断 student **现阶段能力边缘**在哪，造"能做到但还没做熟"的关（p≈0.5 那种）。不追深度，追**适配**。
- **产出倾向**：贴近 student 当前 frontier 的成就、单步或浅前置、巩固性关卡。
- **存在理由**：Critic 只能**否决**太难的，**不能生产**适配的；若无此 proposer，池子会系统性偏难（呼应 [[stage4-realcause-curriculum-too-hard]] 旧教训）。

### 2.3 Proposer-Breadth（生产，往广）
- **侧重**：铺**多样技能类别**（战斗 / 采集 / 制作 / 探索四大类，对应 DiCode Figure 3），避免课程只在一条链上钻。
- **产出倾向**：跨类别覆盖，补 archive 里稀疏的成就类别。

### 2.4 Critic-Feasibility —— 【已判定：冗余，不实现（2026-07-01 代码核实）】
> **原设计**（保留供追溯）：Critic 读[所有提案 + archive 覆盖 + student 状态]，**把关**（否决不可解/太超前/覆盖不新颖）+ **补盲**（主动追加 archive 缺失的垫脚石）。
>
> **判定：Critic 的每一项职责都已被现有 3 persona + 四项 bid 覆盖，硬加会被淹没（[[difficulty-bid-drowned-equals-learnability]] 类教训）。故不实现。** 逐项核实：
>
> | Critic 职责 | 已被谁覆盖 |
> |---|---|
> | 否决**不可解** | Learnability bid：Craftax 里"不可解"= 缺前置 → 训练 p≡0 → learnability=p(1-p)=0 → priority_score 沉底、不被选/激活。DiCode 靠"训练发现 p=0 淘汰"过滤，无需事前判可解性（代码里本就无 solvable/reachable 检查，`check_compilation` 只验代码能 reset+step，不验能否通关）。 |
> | 否决**太难/太超前** | Learnability bid（奖励 p≈0.5，压制 p→0）|
> | 否决**覆盖不新颖** | Coverage bid（次模，重复关的边际增益→0）|
> | **补盲：追加缺失垫脚石** | **Proposer-Ambitious**：其 mandate 就是"识别通向深层成就的 prerequisite chain，把缺失的中间前置 SCAFFOLD 进世界"（persona_ambitious.py:41,120-121）——正是补盲。|
>
> **关于 novelty 卖点**（[[dicode-critic-not-future-work-novelty-boundary]]）：DiCode 的公开难题（区分垫脚石 vs 干扰关）由 **Coverage（论域=67 成就的互补覆盖）+ Ambitious 的前置链推理** 共同回应，不需要一个独立 Critic 角色。
>
> **DiCode 的 `interestingness_critic` 是死配置残桩**：虽作为 LLM 角色被配置传入（sbatch `interestingness_critic=deepinfra`），但 gen_manager 从不调用它（代码零匹配）。`network.py` 的 `ActorCriticRNN` 是 PPO value critic，与此无关。

## 3. 四项 bid（v2 的选择准则）

```
bid_marginal(x | 已选) = w_cov · Coverage(x | 已选)      ← 次模，代码算（加权集合覆盖，depth权重）
                       + w_end · Endorsement(x)          ← 模，LLM 互评聚合
                       + w_amb · AmbitionGain(x)          ← 模，代码算（深度 × target缺口）
                       + w_lrn · Learnability(x)          ← 模，rollout 实测（★新增）
```

### 3.1 谁负责哪一项（重要事实：只有 Endorsement 用 LLM）
| bid 项 | 负责方 | 用 LLM |
|---|---|---|
| Coverage | 纯代码（集合覆盖 + depth 权重） | ❌ |
| AmbitionGain | 纯代码（profile→target_gap × depth） | ❌ |
| **Learnability** | **短 rollout 实测 p**（GPU，非 LLM） | ❌ |
| Endorsement | 三个 proposer 互评（LLM） | ✅ |

### 3.2 ★Learnability 项：为什么加、怎么算【已改为官方做法 2026-06-30】
- **为什么加**：auction 原本三项全偏"深/难"，求稳关被系统性淘汰（§0-3）。Learnability 是**唯一奖励"适配当下"的力量**，与 depth 对冲，给 Proposer-Feasible 撑腰。
- **★重大修正（用户 2026-06-30 提问揭穿）**：原设计写"对候选关跑 256局 rollout 测 p"是**反模式**。DiCode baseline 的 learnability **不做额外 rollout**：`p` 是**训练副产品**（[scoring.py:148](dicode_src/src/dicode/scoring.py#L148) `sr = num_successes/num_finished` 从训练 trajectory 统计），learnability = `sr·(1-sr)` 存进 archive node（[training.py:386](dicode_src/src/dicode/training.py#L386)）。**只有训练过的关有 p，全新候选关没 p**。给未训练的候选关额外测 p 既工程重（撞 worker 线程/GPU 争抢/OOM）又语义错（不符 DiCode 的 p 定义）。详见 [[dicode-learnability-p-is-training-byproduct]]。
- **官方定义（现实现）**：`Learnability(x) = parent_lrn[x.parent_task_id]`，其中 `parent_lrn` = 候选关**父关**在 archive 里已存的官方 learnability `p(1-p)`（`learnability_score`，回退 `priority_score`）。父关在可学区（p≈0.5，learnability≈0.25）→ student 正处这条链的可学边缘 → 子关大概率适配。**纯 archive 读，零 GPU、零 rollout**。父关无 p（如 seed）→ 该项=0，只靠其他三项。
- **数学性质**：per-proposal 独立（父关 learnability 是固定 map）→ **模** → 加进 bid 不破坏次模性 → (1-1/e) 保证仍成立。clamp 到 [0,0.25]。
- **它不是回到 learnability 老路**：v1 §7 分析过 learnability 是"selection-over-existing"、被动。v2 里它有**新角色**——多力量 bid 里的**平衡项**，专门对冲 depth 偏向，不是主信号。

### 3.3 ~~短 rollout 测 p~~【已作废，改官方做法见 §3.2】
原设计（256局并行、完整 step、当前权重不训练）已被官方做法取代——p 不额外测，是训练副产品，Learnability 用父关已存的官方 `p(1-p)`。本节保留仅为记录演进；**实现不含任何额外 rollout**。

### 3.4 ~~rollout 插入时机~~【已作废，改官方做法见 §3.2】
原"全选 36 关串行分批错峰 rollout 测 p"的方案（连同为它做的架构调查：worker 线程池/CPU 隔离/空窗/真并发显存探针）**全部作废**——它们建立在"要给候选关额外测 p"的错误前提上。官方做法下候选关无独立 p，Learnability 走父关代理，**无编译、无 rollout、无显存/线程问题**。
- 历史记录：曾摸清 auction 在 worker 线程（`ThreadPoolExecutor`，无 train_state/GPU），`check_compilation` 只跑 CPU；曾写真并发显存探针待测 L40S 是否装得下 train+rollout。这些随官方做法落地而不再需要。

### 3.5 权重 w_cov/w_end/w_amb/w_lrn
- 开发期固定（如 1/1/1/1），正式做敏感性 ablation。
- **⚠️用户提醒**：上机后必须看 wandb 里四项**实际贡献占比**，防某项被淹没（[[difficulty-bid-drowned-equals-learnability]] 旧教训：difficulty 被 anchored/cenie 淹没到 8%）。四项等权时 depth 出现在 Coverage+Ambition 两项、又被 Learnability 对冲，占比动态须实测。

## 5. 未接 / 待定（诚实清单）
- **v2 价格机制（Walrasian）/ contest**：v1_experiment §7.6-7.7 设计好、单测过，**未接**。当前仍 v1 GreedyTopK。用户问过为何不接——答：我按"先 v1 后 v2"推进但未与用户确认此优先级。**优先级待用户定。**

- **GPU OOM（job 3582377）——已定位（2026-06-30）**：
  - **真因 = 运气分卡，非 auction bug**。C 档与 baseline 的 sbatch 资源请求**逐字节相同**（`--gres=gpu:1` 不指定卡型、`--mem=32G`）。差别只有物理卡：baseline(3575920)→gpu3004=**L40S(48GB)** 扛住；C 档(3582377)→gpu2101=**RTX 3090(24GB)** 炸。
  - **OOM 位置 = seed training 首步** `ppo_tr.py:1037 train_jit`，此时 proposer/cross-rating/auction **一行都没执行** → 与 auction 无关。
  - **数学**：`.err` 显示 remat 压不下 `15.35GiB` + 单次请求 `18.83GiB`，峰值 ~35GiB。Oscar GPU 显存清单：**L40S=48GB（唯一能扛）；RTX 3090 / A5000 / A5500 / Quadro RTX6000 全是 24GB（必炸）**。峰值来自 PPO 反向 + GTrXL（num_envs=1024×num_steps=128、embed 256、window_mem 128、优化器状态）。
  - **修法（不降配置，保 48.33 可比）**：sbatch 加 `#SBATCH --constraint=l40s` 强制只上 L40S。代价 = L40S 少、排队更久。**降 num_envs/timesteps 不可比，禁用。**
- **全选 vs 两阶段 rollout：已定为全选（用户 2026-06-30 拍板）**。见 §3.4 更新。
