# 方法设计 v1 —— 多 FM 竞价课程生成（2026-06-30）

> 工作名（暂）：**Auction-DiCode** / **多 FM 竞价课程**（正式名待定）。
> 基于：new_direction.md 全部已落盘决策 + DiCode 全文精读（机制嵌入点精确到式号）+ 用户 2026-06-30 三项设计拍板。
> 一句话：**把 DiCode 的单 FM "dream 下一关" 换成 N 个异质 FM 各 dream 一关 + LLM Critic 主动补盲 + auction 选互补的 top-k，注入同一 PLR/archive 骨架。**

## 0. 用户已拍板的三个设计决策（本设计的硬约束）

1. **bid 信号 = 选项一（多 FM 分歧/互补覆盖）+ 选项二（LLM Critic 主动 ambitious 提案）组合**。
   - 用户关键纠正（接受）：learnability `p(1-p)` 只是**被动筛选器**（从已生成关里挑 p≈0.5），**不产生新结构、不提多样性**；而 **LLM Critic 的 ambitious 维度是主动的**——能指出 archive 缺某条依赖链并促成造它。**二者本质不同非重叠**：DiCode learnability = selection over existing；我 = generation of novel structure。
2. **选择机制 = 每轮选 top-k 注入**（复用 DiCode PLR/archive 骨架，改动最小，易对标）。完整经济学 auction（预算/价格/博弈）留 future work。
3. **Proposer 异质性 = 两者都做（对照）**：主结果 C 档=异质底座；A 档=同模型×persona 多样性。正好落进 B<A<C 实验骨架。

## 1. 总体框架：DiCode 4 步生成循环里，我改哪一步

DiCode 每个 generation cycle 4 步（§3.2）：**①选 parent → ②生成 NL 描述 → ③生成代码 → ④编译检查**。我的改动**集中在 ②，其余全部复用**：

| 步骤 | DiCode（单 FM） | 我的方法 | 改动 |
|---|---|---|---|
| ① 选 parent | 从 archive 按式7 learnability 选 1 个 parent（式6 限定 A/B 档且子代全 D） | **复用**（同一 archive 有向图 + 式6/7 选 parent） | 不改 |
| ② 生成描述 | **1 个 FM** 出 1 个描述 h（式8） | **N 个异质 Proposer 各出 1 个描述 h_i + LLM Critic 补盲 → auction 选 top-k** | **★核心改动** |
| ③ 生成代码 | 第二个 FM 把描述→代码（式9） | **复用**（Coder 把选出的 k 个描述各→代码） | 不改（k 份并行） |
| ④ 编译检查 | 跑短 trajectory 过滤编译/运行错（不自修正） | **复用** + 加可解性检查（flood-fill/BFS，接旧 generator-solvability 经验） | 微增 |
| 训练 | 20% target + 新生成 + PLR 回放 archive（式2） | **复用**（注入的 k 关进 training batch + archive） | 不改 |
| reward | `r_λ = r_target·I_init + I_success·B_t`（式4，goal bonus B_t 式5 自适应） | **复用**（同 reward 结构；订正上轮"完全不改reward"——DiCode 给生成关加了 goal completion bonus，我继承它，仍不设计新 reward） | 不改 |

**→ 我的贡献被精确隔离在"②描述生成层"**：把"单 FM 独白"换成"多 FM 竞价市场 + 主动补盲 Critic"。这让 ablation 极干净（N=1 且去 Critic ≈ DiCode），且复用 DiCode 全套成熟骨架（archive/PLR/reward/训练）= 站在巨人肩上、易对标。

## 2. ②描述层的精确机制（本方法的灵魂）

给定 ①选出的 parent λ_p（含 perf_p）+ target 成绩单 perf_target，描述层产出 k 个新 level 描述：

### 2.1 Stage A — N 个异质 Proposer 各出一个提案
每个 Proposer_i（异质底座或异质 persona）独立接收 [域上下文 c1 + parent 描述 λ_p + perf_p + perf_target + mutation 指令 m1]（= DiCode 式8 的 context），各自 dream 一个描述 h_i。
- **关键：N 个提案彼此独立、不共享中间结果** → 它们的分歧/互补是真实的（这是单 FM 结构上拿不到的信号）。
- 产出：{h_1, ..., h_N}，每个 h_i 附带 Proposer 自报的"这关教什么技能/攻哪条依赖链"的结构化标签（喂给后续覆盖度计算）。

### 2.2 Stage B — LLM Critic 主动补盲（用户坚持的关键创新）
一个 Critic agent 读 [当前 archive 的技能/依赖链覆盖摘要 + perf_target + {h_i}]，做两件事：
- **(主动) 补盲提案**：若发现 archive 缺某条 target 需要但没人覆盖的依赖链（如"先造火把照明才能深挖"），**主动追加一个 ambitious 描述 h_{N+1}**。← 这是"generation of novel structure"，learnability 筛选器做不到。
- **(护栏) 多维评审**：给每个 h_i 打 [可解性、是否太超前(可学性)、覆盖新颖性] 标签，**砍掉不可解/可学性≈0 的提案**（护栏，非主 bid）。

### 2.3 Stage C — auction 选 top-k（bid = 互补覆盖 + 互相认同）
对存活提案集 {h_i}（含 Critic 补的 h_{N+1}）计算 bid，选 top-k：

```
bid(h_i) = w_cov · Coverage(h_i | 已选集 ∪ archive)     # 互补：补了多少别人没覆盖的依赖链/成就
         + w_end · Endorsement(h_i | {其他 Proposer})    # 互相认同：其他 FM 给它的赞同票
         + w_amb · AmbitionGain(h_i | perf_target)        # 野心：攻的依赖链深度 × target 上该成就的缺口
```
- **Coverage**：贪心增量——已选 j 个后，第 j+1 个按"边际新覆盖"算，保证选出的 k 个互补而非趋同（这是 selection 层防 mode-collapse 的关键）。
- **Endorsement**：每个 Proposer 给其他人提案投票/打分（一轮 cross-rating），被多数认同的得分高 → 过滤掉单个 FM 的怪异提案。**这是真正的"多 FM 市场"信号，单 FM 没有。**
- **w 权重**：开发期固定（如 1/1/1），正式可做敏感性 ablation。
- 选出 top-k → 送 ③ Coder 各生成代码。

**为什么这套 bid 与 DiCode/GenEnv 的 learnability 正交（用户论点的形式化）**：
learnability `p(1-p)` 作用在**单关的成功率标量**上，是 selection-over-existing；我的 bid 三项全部作用在**关之间的关系**上（互补覆盖=关际差异、认同=关际共识、野心=关对 target 缺口的针对性）。**前者无法产生"archive 里没有的结构"，后者主动促成它。** 这就是 §1 命门"bid 须与 learnability 正交"的落地。

## 3. 三档对照实验（对接 new_direction.md §14 的 B<A<C）

| 档 | 描述层配置 | 隔离的变量 |
|---|---|---|
| **B = DiCode 复现** | 1 Proposer，无 Critic，无 auction（=式8 原样） | baseline 擂台（48.33） |
| **A = 同模型 auction** | N×同一模型(persona 异质) + Critic + top-k auction | **机制本身的增益**（模型固定=DiCode同款） |
| **C = 异质 auction** | N×异质底座(Qwen+DeepSeek+GLM) + Critic + top-k auction | **+异质底座的额外增益** |

**额外消融（隔离每个零件）：**
- A 去 Critic：测"主动补盲"单独贡献。
- A 去 Endorsement（bid 只剩 Coverage）：测"多 FM 互相认同"是否必要。
- A 把 top-k 的 Coverage 换成随机选 k：测"互补选择"vs"多提案本身"。
→ 这组消融直接回答"赢在哪个零件"，挡审稿人"哪部分真有用"。

## 4. 立论链（写 intro/motivation 用）

1. **病**：Craftax 后期成就信用分配悬崖，纯 RL/UED 卡 0%（DiCode 已证）。
2. **DiCode 的解 + 局限**：单 FM dream 课程有效，但①单点语义偏差②无外部校验易生成干扰关③覆盖不到的依赖链是永久盲区（DiCode §6 open challenge：难区分有用垫脚石 vs 干扰）。
3. **我的解**：多异质 FM 竞价市场——分歧/互补覆盖 + 互相认同当 bid（单 FM 拿不到的信号）+ Critic 主动补盲（learnability 筛选器做不到的 novel structure generation）。
4. **证据**：B<A<C 递进 + 零件消融 + 超 48.33。

## 5. 待定/风险（设计阶段需继续敲定）

- **[待定] N 和 k 取值**：N=3 起步（对应三异质底座）；k 待定（DiCode 每 cycle 生成多少关？需回 DiCode 附录查 batch size 对齐）。
- **[待定] Coverage 的"依赖链/成就覆盖"如何具体度量**：用 Craftax 22+ 成就的 multi-hot？还是 Proposer 自报标签？需可计算、可复现。
- **[待定] Endorsement 的 cross-rating 成本**：N 个 Proposer 互评 = N×(N-1) 次额外 LLM 调用/轮 → 接 auction 成本论证（待办3）。N=3 时 =6 次/轮，可接受。
  - **[2026-06-30 实测更新]** auction 成本论证现有真实数据支撑（见 §7）：DeepInfra 在 12 并发时已是 **5.6× 半并发**（非真并行），故 N-Proposer 把并发数从 12→12N 会加剧排队。这是成本论证必须正视的点——缓解方向：①N-Proposer 分批控并发；②正式跑用真并发端点（Fireworks/dedicated）；③Endorsement 的 N×(N-1) 互评压在短 prompt。
- **[风险] A 档 persona 多样性可能太弱**（同模型提案趋同）→ 若 A≈B，则"机制有用"证据弱，需保证 persona prompt 差异足够大 + 温度够高。
- **[风险] 实现载体**：DiCode 代码开源（github.com/konstantinosmitsides/dreaming-in-code）→ **应基于它改 ②描述层，不重写**。需先跑通 DiCode 复现（=B 档）再加我的层。
- **[待核] DiCode 是否真有 reward 改动**：式4 的 goal bonus B_t 确认 DiCode 给生成关加了 completion bonus（非纯原生 reward）——我继承它，但这意味着"我不设计 reward"的表述要精确为"我沿用 DiCode 的 reward 结构，不引入新 reward 项"。

### 7.3 对方法论证的两个影响
1. **★efficiency 卖点（新增可写论据）**：换商业弹性端点后，完整复现单 seed 估算 **~12-13h**（FM ~2.1h + GPU 训练 ~10.5h），远低于作者 **48h**，且**一个 48h 墙钟 job 能跑完单 seed**（不必 checkpoint 续跑）。**注意口径**：return 数值必须与 DiCode 同口径对标（48.33/1024 held-out/5 seed），wall-clock 加速是**额外卖点非替代**。作者用 48h 非认知局限（学术免费校集群 vs 商业付费的工程选择，且固定基础设施利于复现性）——我愿花小钱换真并发是合理优化。
2. **★N-Proposer 成本论证（§5 待办3 落地）**：DeepInfra 在 12 并发已 5.6× 半并发（开始排队），N-Proposer 把并发 12→12N 会加剧。缓解：分批控并发 / 正式跑用真并发端点（Fireworks/dedicated）/ Endorsement 互评压短 prompt。这反向支持一个论点：**N-Proposer 多 FM 架构天然更吃"真并发"端点，是 architecture-infrastructure fit**（真并发端点上 12N 几乎不增 wall-clock）。
